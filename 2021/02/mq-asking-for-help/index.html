MQ: Asking for help.

Posted on Sat 13 February 2021 in MQ

MQ messaging application trouble shooting: Asking for help.

Whether you are new to IBM MQ or new to the application you are supporting, this discussion is intended to help give you the foundation to reach out for help in the cases where you need it. This information is targeted at those who are new at supporting an application that uses MQ.

Every now and then in your career as a developer you may have to support an application that uses IBM MQ to communicate with other applications. Most likely the reason this application has come to your attention is that perhaps it is having an issue of some sort. These issues at first are usually categorized into some over simplified group and then upon looking into more detail they fall into some other common categories.

The purpose of these notes are to give you some language/tools to discuss the issue with support professionals that work with IBM MQ.
You may encounter a variety or experience levels and in some cases you will need to bring in a number of groups to get to the bottom of the performance issue. Hopefully these notes will help you get to the resolution more quickly.

Problem Categorization and category evolution:

The problem usually starts out as "We have a MQ problem". Upon further discussion it usually falls into one of these Categories:

  1. Lost message. ->Not really lost, just not where it was expected.
  2. Not getting a response. ->Messaging servicing application had a problem with the request.
  3. Not getting a response on-time. ->Everything appears to work, just not fast enough. These issues can be challenging.

Helpful information to gather prior to contacting MQ Team:

Calling in the MQ team and saying "we have a MQ problem" and having no additional information may not get you very far. If you need help you have to ask, so reaching out for help is encouraged. The first time you do this may not go as well as you hope, but with knowledge and experience trouble shooting will become easier.

It is understood that an application support person may not always know all the details of what infrastructure is being used. Perhaps these details are buried under ten layers of abstraction or in any number of configuration files. You could be new to the application. The application deployment group could be seperate from the support group. Any number of reasons could contribute to this lack of information. At some point these details will likely be needed. It would be good to know how to qet these details if you are going to be supporting the application.

What information to have prior to contacting the MQ group?

  1. Hostname and TCP/IP port.
  2. queue manager and queue name(s).
  3. Time the problem was first noted.

In some cases host name and port would not be used if you use local bindings. In which case queue manager name and queue will be all you have. Local bindings is not that common, if you use them your application and the MQ you connect to would reside on the same host. If you have contacts for the team that supports the application that services your message that can prove very helplful.

Is there anything I can ask of the MQ support group if I don't know the Hostname, port, queue manager and queue?

There are questions you can ask that may be revealing, however even if this information can be provide by the MQ support group it may not be conclusive.

Questions to ask MQ support when you don't know underlying MQ infrastructure that supports your application:

  1. Are any queue managers down?
  2. Are there messages on dead-letter queues?
  3. Are there messages on transmit queues?

If messages started building up on either transmit or dead letter queues at the same time that the application started having issues this may be a good indicator that these events are related.

Something to consider:

One thing to note about this is that if queue managers are down for maintenance/patching or if there are transmit queue backups this doesn't mean that your application is affected by this. If the problem goes away after the queue manager is back active or the transmit queues clear, that may be a good indicator it was related, but without knowing what infrastructure your application uses, you won't know for certain.

Once the MQ environment is back to a fully operational status, issues can clear up. If your issue is resolved without you ever determining the details of what infrastructure you use, it would still be wise to learn the details. If you need an environment that is operational during patching/maintenance windows there are ways to accomodate this requirement from a MQ configuration perspective. The backend application servicing the MQ message may have requirements beyond a HA(Highly Available) MQ configuration.
